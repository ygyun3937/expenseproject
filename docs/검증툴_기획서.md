# 경비 정산 검증 툴 기획서

**문서 버전**: v1.0
**작성일**: 2026-04-14
**작성**: AX 혁신실

---

## 1. 배경 및 목적

### 1.1 배경
- 현재 MAK 경비 정산 시스템(v26.04)은 AI 기반으로 영수증을 자동 인식하여 정산서를 작성
- 회계팀에서 제출된 정산서의 **적정성·정확성·규정 준수 여부**를 수작업으로 확인 중 → 시간 소요·누락 위험
- 정산서 **생성** 은 자동화됐지만 **검증** 은 여전히 수동

### 1.2 목적
- 회계팀이 제출된 정산서(엑셀/PDF/JSON)를 업로드하면 **회사 규정 위반·계산 오류·누락 항목** 을 자동 검출
- 검증 리포트(합격/불합격 + 사유) 출력 → 결재 전 1차 필터링
- 향후 ERP 연동으로 확장 가능한 구조

## 2. 기능 요구사항

### 2.1 필수 검증 항목 (1차 릴리스)

| # | 검증 항목 | 규정/기준 |
|---|---|---|
| 1 | 식비 한도 | 1인당 12,000원 이하 (초과 시 한도까지만 반영되었는지 확인) |
| 2 | 식비 인원수 증빙 | 한도 초과 건에 `[한도적용 원금 XX,XXX]` 메모 존재 여부 |
| 3 | 카테고리-금액 일관성 | 주차비 영수증이 교통비 칼럼에 들어가는 등 매핑 오류 |
| 4 | 합계 검산 | 항목별 합계 = 각 행 합. 개인경비 합계 + 출장일비 = 총계 |
| 5 | 유류비 공식 검증 | 일반: `거리 ÷ 연비 × 유가`, EV: `기본료 + (거리 × 전력량단가 ÷ 복합전비)` |
| 6 | 유류비 날짜 규칙 | 1차 = 해당 월 15일, 2차 = 전달 말일 |
| 7 | 출장일비 공식 | `일수 × 30,000원` (국내) / 지역·직급별 기준(해외) |
| 8 | 환율 출처 | 해외 경비 환율이 서울외국환중개 매매기준율과 ±1% 이내 |
| 9 | JPY 100엔 기준 | 일본 엔화 환율 기입 시 1엔 기준 환산되었는지 |
| 10 | 중복 영수증 탐지 | (일자 + 금액 + 상점명) 동일 건 2회 이상 제출 여부 |
| 11 | 필수 필드 | 성명·부서·작성일·정산차수 기입 여부 |
| 12 | 증빙 번호 연속성 | 증빙 seq 1,2,3... 중간 누락·중복 |

### 2.2 확장 검증 (2차 릴리스)
- 영수증 이미지 **재스캔** 후 원본 금액과 기입 금액 대조 (OCR 재검)
- 거래처 접대비 건 **참석자·거래처 기입** 필수 체크
- 주말·공휴일 결제 건 이상치 플래그
- 개인별 월 한도 집계 (예: 1인 월 식대 누계)
- 부서 예산 대비 사용률

### 2.3 출력물
- **검증 리포트 (HTML/PDF)**
  - 항목별 ✅/⚠️/❌ 요약
  - 위반 건 상세 (어느 행·어느 금액·어느 규정)
  - 원본 정산서 스크린샷 + 하이라이트
  - 합격/반려 판정 (중대 위반 1건이라도 있으면 반려)
- **CSV 에러 로그** (ERP 업로드용)
- **관리자 대시보드** (일자별 검증 건수·반려율)

## 4. 기술 스택

### 4.1 Backend
| 레이어 | 선택 | 이유 |
|---|---|---|
| 언어 | **Python 3.12** | 기존 정산 시스템과 동일 스택 → 규정 로직 재사용 가능 |
| 웹 프레임워크 | **Flask** (→ FastAPI 확장) | 기존 경비 시스템과 동일, 점진 마이그레이션 용이 |
| 검증 엔진 | Pydantic + 커스텀 Rule Engine | 스키마 검증 + 규정 체크 모듈화 |
| PDF 파싱 | **PyMuPDF (fitz)** | 기존 시스템과 동일 |
| Excel 파싱 | **openpyxl** | 회계팀이 엑셀 제출할 경우 |
| OCR 재검 (2차) | Google Gemini 2.5 Pro / Claude Sonnet | 기존 시스템 라이센스 재활용 |
| 환율 조회 | 서울외국환중개 스크래퍼 (기존 `exchange_rate.py` 재사용) |  |

### 4.2 Frontend
| 레이어 | 선택 | 이유 |
|---|---|---|
| UI | **HTML + Tailwind CSS** (1차) → React (2차) | 1차는 경량 SSR, 이후 인터랙션 확대 시 React |
| PDF 뷰어 | PDF.js | 원본 정산서 인라인 렌더 |
| 리포트 출력 | HTML → 브라우저 인쇄 PDF (1차) / WeasyPrint (2차) |

### 4.3 Infra / DevOps
| 항목 | 선택 |
|---|---|
| 배포 환경 | **사내 Windows 서버** (기존 경비 시스템과 동일), `waitress` + Windows Service |
| DB | SQLite (1차) → PostgreSQL (2차, 이력 누적 시) |
| 인증 | LDAP / 사내 SSO |
| 로깅 | Python `logging` + 파일 로테이션 |
| CI | 1차: 수동 배포 / 2차: GitHub Actions + WSL 빌드 |

### 4.4 규정 엔진 설계
- 각 규정을 **독립 모듈**로 구현 (`rules/meal_limit.py`, `rules/fuel_formula.py`, `rules/exchange_rate.py` ...)
- Rule 베이스 클래스
  ```python
  class Rule:
      id: str
      name: str
      severity: "error" | "warning" | "info"
      def check(self, report: ExpenseReport) -> list[Violation]: ...
  ```
- 규정 변경 시 해당 모듈만 수정 → 배포 영향 최소화
- 규정 활성/비활성 토글 (설정 파일)

## 5. 데이터 흐름

### 5.1 입력 포맷
1. **JSON** — 현 경비 시스템의 임시저장 데이터 export 기능 추가해서 제출
2. **PDF** — 인쇄된 정산서. PyMuPDF로 테이블 추출
3. **엑셀** — 회계팀 양식

### 5.2 내부 표현 (Canonical Model)
```python
class ExpenseReport(BaseModel):
    writer: Writer         # 성명, 부서, 작성일, 차수
    period: SettlementPeriod
    expenses: list[ExpenseItem]      # 개인경비
    trip_allowance: list[TripItem]   # 출장일비
    fuel_log: list[FuelLogItem]      # 유류비 주행 로그
    overseas: list[OverseasItem]
    totals: Totals
```

### 5.3 검증 흐름
1. 입력 파일 파싱 → `ExpenseReport` 객체
2. 활성화된 모든 Rule을 순차 실행 → `list[Violation]` 수집
3. severity 기준으로 집계 → 합격/반려 판정
4. 리포트 생성 (HTML 템플릿 + 위반 목록)
